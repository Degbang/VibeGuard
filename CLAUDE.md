## 2. What VibeGuard Is

A five-layer explainable security assessment framework for Java microservices generated by AI coding tools (Copilot, Cursor, ChatGPT, Bolt, etc.). It detects, scores, and explains vulnerabilities across five CWE categories, using a hybrid of rule-based detection and machine learning, with SHAP-based explainability so results are interpretable, not a black-box score.

**Do not conflate these two things** (a documented point of confusion to avoid):
- SHAP explains **feature-level attribution** вЂ” why the ML model scored something the way it did.
- Layer 1 (AST parsing) provides **file/line traceability** вЂ” where in the source the issue physically is.
These serve different purposes and both must be present in final output.

## 4. Coding Standards (apply to every file, every layer)

### Structure & typing
- Full type hints on every function signature and every class attribute.
- Use `@dataclass` (or `pydantic` models if validation logic grows complex) for structured data вЂ” never raw dicts passed between modules.
- One responsibility per function. If a function needs "and" in its description, split it.
- No function should exceed ~40-50 lines as a soft ceiling; if it does, it's a signal to decompose.

### Documentation
- Docstrings on every public function, class, and module вЂ” Google-style docstrings, applied consistently across the whole codebase.
- Every module starts with a short module-level docstring stating its role in the five-layer pipeline.

### Style & tooling
- PEP 8 compliant.
- **Formatter:** `black`, run before every commit.
- **Linter:** `ruff` (covers flake8-equivalent checks faster). Zero lint warnings in committed code.
- **Type checker:** `mypy` in strict-ish mode (`disallow_untyped_defs = true` at minimum) вЂ” this is a security tool; type errors are bugs waiting to happen.
- Add all three to `requirements.txt` / `pyproject.toml` from the very first commit, not retrofitted later.

### Logging & errors
- Use Python's `logging` module. Never `print()` for anything other than final user-facing CLI/report output.
- Fail closed, never silently: a file that can't be parsed, scored, or classified must be explicitly flagged in output (`parse_failed`, `scoring_failed`, etc.), never dropped from results without a trace. Silent failures in a security tool under-report vulnerabilities, which is a correctness bug, not a cosmetic one.

### Testing
- `pytest` for everything. Every layer gets its own `tests/` coverage before being considered "done."
- Maintain fixture Java files under a dedicated `tests/fixtures/` directory: at minimum, one clean file, one file per target CWE with a known instance of that vulnerability, one malformed/unparseable file, one empty file.
- Target meaningful coverage on detection logic specifically (rule modules, scorer, ML pipeline) вЂ” coverage on glue code matters less than coverage on the parts making security judgments.

## 5. Security Requirements (this tool analyses untrusted, AI-generated code)

- **Never execute or `eval()` any content from a target Java file.** VibeGuard only ever parses text into an AST вЂ” it must never run the code it's analysing. State this explicitly in a comment in `scanner.py` and in the thesis, since a reviewer will ask.
- Guard against path traversal: resolve all file paths with `Path.resolve()` and verify they remain inside the expected sample-apps root before reading.
- Set limits on input size and parse time per file (a large or adversarial file must not hang or OOM the process) вЂ” this is also a legitimate "robustness" evaluation point for Chapter 4/5.
- No secrets, API keys, or credentials вЂ” real or plausible-looking вЂ” committed anywhere outside clearly labeled test fixtures.
- Dependency hygiene: pin versions in `requirements.txt`, and periodically check for known-vulnerable dependencies (there's a nice irony/rigor point in a security thesis project itself passing a dependency audit вЂ” worth a line in Chapter 5 if true).

## 7. Build Order

Build and validate one layer at a time. Do not start Layer 2 work until Layer 1 (parser + all five CWE rules + scanner) is tested and working end to end. This mirrors the thesis's own Chapter 4 structure, so implementation order and write-up order stay in sync.

1. `ast_parser.py` вЂ” Java AST parsing via `javalang`, structured `ParsedFile`/`ParsedClass`/`ParsedMethod` output.
2. `rules/cwe_798.py` вЂ” hardcoded secrets detection (first rule, validates the parser's string-literal extraction).
3. Remaining four CWE rule modules.
4. `scanner.py` вЂ” orchestrates parser + all rule modules across a directory of sample apps.
5. Layer 2 (feature extraction) в†’ Layer 3 (rule-based scoring) в†’ Layer 4 (ML) в†’ Layer 5 (SHAP explainability + reporting).
6. `evaluation/evaluate.py` last, once there's something to evaluate.
